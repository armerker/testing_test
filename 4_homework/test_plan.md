## 1. Введение

### 1.1 Цель тестирования
Проверка корректности взаимодействия между Сервисом А (REST API) и Сервисом B (AMQP) при управлении услугами (подключение, отключение, блокировка). Особое внимание уделяется упорядочиванию команд, последовательной обработке и атомарности изменения статуса.

### 1.2 Область применения
Документ предназначен для команды тестирования и разработки. Определяет объем, подход, ресурсы и график тестирования взаимодействия двух сервисов.

### 1.3 Ссылки на исходные документы

- Спецификация REST API Сервиса А
- Спецификация AMQP протокола взаимодействия
- Архитектурная документация

## 2. Объект тестирования (Что надо тестировать?)

### 2.1 Компоненты системы

| Компонент | Описание | Интерфейсы |
|:----------|:---------|:-----------|
| **Сервис А** | Принимает REST запросы, управляет очередью команд, обеспечивает упорядочивание | REST (HTTP), AMQP (Producer/Consumer) |
| **Сервис B** | Выполняет операции подключения/отключения/блокировки, возвращает результат | AMQP (Consumer/Producer) |
| **Очередь AMQP** | Транспорт для сообщений между сервисами | RabbitMQ / ActiveMQ |

### 2.2 Типы операций

| Операция | REST запрос | AMQP команда | Статус услуги |
|:--------:|:------------|:--------------|:---------------|
| Подключение | POST /activate | `{ "cmd": "activate", "serviceId": "123" }` | "активна" |
| Отключение | POST /deactivate | `{ "cmd": "deactivate", "serviceId": "123" }` | "неактивна" |
| Блокировка | POST /block | `{ "cmd": "block", "serviceId": "123" }` | "заблокирована" |

### 2.3 Правила взаимодействия
- Сервис А принимает REST запросы независимо от состояния Сервиса B
- Команды отправляются в Сервис B строго последовательно (не более 1 команды в ожидании)
- Следующая команда отправляется только после получения ответа на предыдущую
- Статус услуги изменяется ТОЛЬКО после получения положительного ответа от Сервиса B
- Упорядочивание команд сохраняется для всех типов операций
- При отрицательном ответе статус услуги не меняется, но очередь продолжает обрабатываться

## 3. Что будете тестировать? (Объем тестирования)

### 3.1 Что тестируемо

Проверке подлежат следующие функциональные области:

| № | Область | Описание |
|:--:|:--------|:---------|
| 1 | Базовые операции | Проверка успешного подключения/отключения/блокировки |
| 2 | Последовательность и упорядочивание | Проверка, что команды выполняются строго по порядку |
| 3 | Обработка ошибок Сервиса B | Отказы, таймауты, невалидные ответы |
| 4 | Независимость приема REST запросов | Прием запросов при занятом или недоступном Сервисе B |
| 5 | Атомарность изменения статуса | Статус меняется только после подтверждения |
| 6 | Восстановление после сбоев | Падение сервисов, потеря соединения |
| 7 | Негативные сценарии | Некорректные запросы, несуществующие услуги |
| 8 | Производительность и надежность | Очереди, нагрузка, потеря сообщений |

### 3.2 Что не тестируемо
- Нагрузочное тестирование самой AMQP инфраструктуры
- Тестирование безопасности (кроме базовой аутентификации REST)
- Клиентские приложения
- Другие сервисы, не участвующие в данном взаимодействии

### Раздел 1: Базовые сценарии (Happy Path)

**Тест-кейс TC-01: Проверка успешного выполнения операций**

**Предусловия:**
- Сервис А и Сервис B запущены и работают
- Очередь AMQP доступна
- Существует тестовая услуга с ID = "123"
- Определен базовый URL Сервиса А (`{{base_url}}`)

**Шаги в Postman:**
1. Открыть Postman
2. Создать новый запрос: **POST**
3. Ввести URL: `{{base_url}}/activate`
4. Перейти во вкладку **Body**, выбрать **raw** и формат **JSON**
5. Ввести тело запроса: `{ "serviceId": "123" }`
6. Нажать кнопку **Send**
7. Проверить HTTP статус ответа
8. Проверить тело ответа
9. Дождаться обработки Сервисом B
10. Выполнить GET запрос для проверки финального статуса

| № | Операция | REST запрос | Тело запроса | Ожидаемый результат |
|:-:|:---------|:------------|:--------------|:--------------------|
| 1.1 | Подключение | `POST {{base_url}}/activate` | `{ "serviceId": "123" }` | HTTP 200, `{ "status": "accepted", "message": "Request queued" }`<br>Финальный статус: `{ "serviceId": "123", "state": "active" }` |
| 1.2 | Отключение | `POST {{base_url}}/deactivate` | `{ "serviceId": "123" }` | HTTP 200, `{ "status": "accepted", "message": "Request queued" }`<br>Финальный статус: `{ "serviceId": "123", "state": "inactive" }` |
| 1.3 | Блокировка | `POST {{base_url}}/block` | `{ "serviceId": "123" }` | HTTP 200, `{ "status": "accepted", "message": "Request queued" }`<br>Финальный статус: `{ "serviceId": "123", "state": "blocked" }` |

### Раздел 2: Последовательность и упорядочивание

**Тест-кейс TC-02: Проверка последовательной обработки команд**

**Предусловия:**
- Сервис А и Сервис B запущены
- Настроена задержка ответа от Сервиса B (1-2 секунды)
- Подготовлены тестовые услуги A, B, C
- Определен базовый URL Сервиса А (`{{base_url}}`)

**Шаги в Postman:**
1. Открыть Postman
2. Создать несколько вкладок с запросами **POST**
3. Для каждого запроса указать соответствующий URL и тело
4. Отправить запросы подряд (не дожидаясь ответов)
5. Проверить HTTP статус каждого ответа (должен быть 200)
6. Мониторить очередь AMQP через Admin UI
7. Проверить финальные статусы через GET запросы

| № | Название | Шаги в Postman | Ожидаемый результат |
|:-:|:---------|:---------------|:--------------------|

Сервис B отвечает на #2 | Ответы на запросы: HTTP 200, `{ "status": "accepted" }`<br>Команда #2 отправлена только после ответа на #1<br>GET статусы: услуга 1 = active, услуга 2 = active |

| 2.3 | Смешанные команды для разных услуг | 1. `POST /activate` (A)<br>2. `POST /deactivate` (B)<br>3. `POST /block` (A) | Все ответы: HTTP 200, `{ "status": "accepted" }`<br>Порядок соблюдается: A-акт → B-деакт → A-блок<br>GET: A = blocked, B = inactive |
| 2.4 | Команды для одной услуги подряд | 1. `POST /activate` (X)<br>2. `POST /deactivate` (X)<br>3. `POST /activate` (X)<br>4. `POST /block` (X) | Все ответы: HTTP 200, `{ "status": "accepted" }`<br>Финальный GET: `{ "serviceId": "X", "state": "blocked" }` |

### Раздел 3: Обработка ошибок от Сервиса B

**Тест-кейс TC-03: Проверка обработки ошибок от Сервиса B**

**Предусловия:**
- Сервис А запущен и работает
- Подготовлен мок Сервиса B с возможностью возврата различных ошибок
- Определен базовый URL Сервиса А (`{{base_url}}`)
- Очередь AMQP доступна для мониторинга

**Шаги в Postman:**
1. Открыть Postman
2. Создать новый запрос: **POST**
3. Ввести URL: `{{base_url}}/activate` (или /deactivate, /block в зависимости от теста)
4. Перейти во вкладку **Body**, выбрать **raw** и формат **JSON**
5. Ввести тело запроса: `{ "serviceId": "123" }`
6. Настроить мок Сервиса B на соответствующий тип ошибки
7. Нажать кнопку **Send**
8. Проверить HTTP статус ответа от Сервиса А
9. Проверить в RabbitMQ Admin UI, что команда ушла в очередь
10. Проверить логи Сервиса А на наличие записи об ошибке
11. Проверить финальный статус услуги через GET запрос

| № | Тип ошибки | Настройка мока B | Ожидаемый результат |
|:-:|:-----------|:-----------------|:--------------------|
| 3.1 | Отказ в подключении | Мок возвращает `{ "status": "error", "message": "Cannot activate" }` | HTTP 200, `{ "status": "accepted", "message": "Request queued" }`<br>Статус услуги не изменен (GET: inactive), очередь продолжается |
| 3.2 | Таймаут ответа | Мок не отвечает 30 секунд | HTTP 200, `{ "status": "accepted", "message": "Request queued" }`<br>Повторные попытки отправки, после N попыток команда помечается ошибочной |
| 3.3 | Невалидный ответ (битый JSON) | Мок отправляет `{ "status": "success", ` (незакрытый JSON) | HTTP 200, `{ "status": "accepted", "message": "Request queued" }`<br>Ошибка логируется, команда не выполнена, статус не изменен |
| 3.4 | Несоответствие команды | На запрос "activate" мок отвечает подтверждением на "deactivate" | HTTP 200, `{ "status": "accepted", "message": "Request queued" }`<br>Ответ игнорируется, команда остается в ожидании |

### Раздел 5: Атомарность изменения статуса

**Тест-кейс TC-05: Проверка атомарности изменения статуса**

**Предусловия:**
- Сервис А и Сервис B запущены
- Сервис B настроен на задержку ответа (2 секунды)
- Определен базовый URL Сервиса А (`{{base_url}}`)
- Есть GET эндпоинт для проверки статуса: `{{base_url}}/status/{serviceId}`

**Шаги в Postman:**
