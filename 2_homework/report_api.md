## 1. Введение

Целью данной работы является тестирование REST API метода создания папки на Яндекс.Диске на соответствие предоставленным требованиям. Тестирование проводилось с помощью инструмента Postman. Скриншоты результатов находятся в папке `screen_api`.

---

## 2. Тестирование требований (Requirements Testing)

В данном разделе проведен анализ исходных требований к API на соответствие критериям качества.

### 2.1. Исходный текст требований

Реализовано REST API

PUT https://cloud-api.yandex.net:443/v1/disk/resources 

Заголовками Content-Type: application/json, Authorization: [token]

Параметры строки запроса:
path - Путь к создаваемой папке,
fields - Список возвращаемых атрибутов

В случае успешного выполнения запроса получен ответ 200, папка появилась по нужному адресу в диске

Если какие-то данные некорректны - ответ 400

При неверном или отсутствующем токене - ответ 401

При попытке создания уже по существующему пути - ответ 400

При возврате ответа от сервера отличному от 200 - папка создана не была, получено тело ответа следующего вида:
{ "message": "string", "description": "string", "error": "string" }


### 2.2. Анализ на основе критериев качества

| Критерий | Оценка | Комментарий |
|:---------|:-------|:------------|
| Полнота | Не полностью | Не указаны все возможные коды ответов (201, 409, 403, 507 и др.). Не описаны ограничения на path (длина, допустимые символы). Не указано поведение с полем fields. |
| Ясность | Неясно | Пункт про ответ 200 противоречит официальной документации API Яндекс.Диска (должен быть 201). Непонятно, какие именно атрибуты можно запросить в fields. |
| Корректность | Некорректно | Утверждение про ответ 200 при успехе содержит ошибку. Также неверно указан код 400 при создании существующей папки (должен быть 409 Conflict). |
| Тестируемость | Тестируемо | Требования можно проверить с помощью Postman или cURL, отправляя различные запросы. |
| Согласованность | Согласовано | Внутренних противоречий между пунктами не обнаружено. |

### 2.3. Вывод по тестированию требований

Требования содержат фактические ошибки (неверные коды ответов) и нуждаются в доработке: необходимо привести коды в соответствие с документацией API, добавить ограничения на входные параметры и описать возможные коды ошибок.

## 3. Тестирование системы (API Testing)

Было проведено тестирование методом `PUT /v1/disk/resources` с использованием Postman. Для тестирования использовался валидный OAuth-токен с правами `cloud_api:disk.write`.

### 3.1. Результаты проверки (соответствие требованиям)

| № | Тест-кейс | Запрос | Ожидание (по заданию) | Реальность | Статус |
|:-:|:----------|:-------|:----------------------|:-----------|:--------|
| 1 | Создание новой папки в корневом каталоге | `PUT /v1/disk/resources?path=/test-folder-1` | 200 OK | 201 Created | Баг |
| 2 | Создание вложенной папки внутри существующей | `PUT /v1/disk/resources?path=/test-folder-1/inner-folder` | 200 OK | 201 Created | Баг |
| 3 | Создание папки без обязательного параметра path | `PUT /v1/disk/resources` | 400 Bad Request | 400 Bad Request | Ок |
| 4 | Создание папки с пустым значением path | `PUT /v1/disk/resources?path=` | 400 Bad Request | 400 Bad Request | Ок |
| 5 | Создание папки с очень длинным именем (256 символов) | `PUT /v1/disk/resources?path=/` + 256 символов 'a' | 400 Bad Request | 404 Not Found | Баг |
| 6 | Создание папки без токена авторизации | `PUT /v1/disk/resources?path=/test-no-token` (без Authorization) | 401 Unauthorized | 401 Unauthorized | Ок |
| 7 | Создание папки с неверным токеном авторизации | `PUT /v1/disk/resources?path=/test-no-token` (Authorization: 12345) | 401 Unauthorized | 401 Unauthorized | Ок |
| 8 | Повторное создание уже существующей папки | `PUT /v1/disk/resources?path=/test-folder-1` (повторно) | 400 Bad Request | 409 Conflict | Баг |

### 3.2. Проверка формата сообщения об ошибке

Согласно требованию: *"При возврате ответа от сервера отличному от 200 - получено тело ответа вида `{ "message": "string", "description": "string", "error": "string" }`"*

| № теста | Код ответа | Наличие всех полей (message, description, error) | Соответствие формату |
|:-------:|:----------:|:------------------------------------------------:|:--------------------:|
| 3 | 400 | Да | Соответствует |
| 4 | 400 | Да | Соответствует |
| 5 | 404 | Да | Соответствует |
| 6 | 401 | Да | Соответствует |
| 7 | 401 | Да | Соответствует |
| 8 | 409 | Да | Соответствует |

**Вывод:** Требование к формату ошибки выполняется полностью. Во всех негативных сценариях тело ответа содержит все три обязательных поля.

## 4. Баг-репорты (Bug Reports)

На основе обнаруженных несоответствий были заведены следующие баг-репорты.

### Баг-репорт №1: Неверный код ответа при успешном создании папки

| Поле | Значение |
|:-----|:---------|
| **Заголовок** | При успешном создании папки API возвращает код 201, что не соответствует требованию (200 OK) |
| **Проект** | Яндекс.Диск API |
| **Серьезность** | Minor (ошибка в документации/требованиях, функционал работает корректно) |
| **Приоритет** | Low |
| **Окружение** | Postman, Windows 11, браузер Chrome |
| **Шаги для воспроизведения** | 1. Отправить PUT запрос на `https://cloud-api.yandex.net/v1/disk/resources?path=/test-folder`<br>2. С заголовком `Authorization: OAuth <валидный_токен>` |
| **Фактический результат** | Код ответа 201 Created |
| **Ожидаемый результат** | Согласно требованию, код ответа должен быть 200 OK |
| **Примечание** | Фактическое поведение API соответствует официальной документации Яндекса (201 Created), но противоречит тестируемым требованиям |
| **Скриншот** | `screen/тест1.PNG` |

## 5. Общий вывод

В ходе работы были протестированы требования и реализованный API метод создания папки.

1. **Требования** содержат фактические ошибки в указании кодов ответов и нуждаются в доработке:
   Успешное создание: должно быть 201 Created, а не 200 OK
   Создание существующей папки: должно быть 409 Conflict, а не 400 Bad Request

2. **API** в целом работает стабильно, но обнаружено три дефекта:
   Несоответствие кодов ответа требованиям (баги №1 и №2)
   Некорректная обработка очень длинного пути (баг №3)

3. **Формат ошибки** соответствует требованию: во всех негативных сценариях тело ответа содержит поля `message`, `description` и `error`.

Дефекты оформлены в виде баг-репортов и требуют исправления в документации (баги №1, №2) и в обработке граничных значений (баг №3).
